# dev-flow — 코드에서 자동 추출하는 계층형 프로젝트 그래프

- 상태: 설계 승인됨 (Phase 1 스펙)
- 작성일: 2026-06-28
- 대상: TypeScript/JavaScript (첫 분석기 플러그인)

## 1. 한 줄 정의

TS/JS 코드를 정적 분석해서 **"기능(진입점) → 프로세스(호출 체인) → 데이터(타입/모델) 흐름"**을
UML 없이 **드릴다운 가능한 인터랙티브 그래프**로 보여주는 도구.

## 2. 목표 (Why)

- 하나의 개발 프로젝트를 그래프로 표현한다.
- 어떤 기능이 어떤 프로세스(호출 흐름)인지 쉽게 확인한다.
- UML 등을 따로 작성하지 않아도, 어떤 기능이 어떤 프로세스·데이터 흐름을 가지는지 이해한다.
- 처음엔 크게(기능 단위) 보고, 자세히 보고 싶으면 노드를 펼쳐(semantic zoom) 내부 함수·모듈
  호출까지 드릴다운한다.

## 3. 핵심 설계 원칙

### 3.1 언어 무관 코어 + 언어별 분석기 플러그인 (LSP / tree-sitter 패턴)

```
┌───────────────────┐    ┌──────────────────┐    ┌────────────────────┐
│  분석기 플러그인     │    │   그래프 IR        │    │   뷰어 (웹 앱)        │
│  (Phase1 = TS)     │───▶│   (JSON, 버전드)   │───▶│  React + React Flow │
│  ts-morph 기반      │    │   = 진짜 코어       │    │  드릴다운/시맨틱줌    │
│  별도 프로세스       │    │   언어 무관         │    │  Tauri webview 호환  │
└───────────────────┘    └──────────────────┘    └────────────────────┘
```

- **진짜 코어 = 그래프 IR(JSON 스키마) + 뷰어.** 둘 다 언어 무관(language-agnostic).
- 언어마다 **분석기 플러그인**을 따로 둔다. 각 플러그인은 그 언어의 네이티브 도구를 쓴다
  (TS = ts-morph, 추후 Python = `ast`, Go = `go/ast`). 모든 플러그인은 **같은 IR을 뱉는다.**
- 분석기를 무슨 언어로 작성하느냐는 각 플러그인의 자유다. TS 분석기를 TS로 작성하는 것이
  전체 시스템을 TS에 묶지 않는다.

### 3.2 분석기와 뷰어를 IR로 분리, 분석기는 별도 프로세스

- 분석기와 뷰어 사이의 유일한 계약은 **그래프 IR(JSON)**.
- 분석기는 별도 프로세스로 둔다. 이 분리가 강제되어야 양쪽을 독립적으로 테스트·교체할 수 있다.

### 3.3 Tauri 확장을 처음부터 고려

- 뷰어는 **처음부터 웹 앱**으로 작성한다.
- **Phase 1**: 분석기 + IR + 웹 뷰어. 브라우저에서 동작. 가치를 빠르게 검증.
- **Phase 2**: 그 웹 뷰어를 **그대로** Tauri webview에 넣고, 분석기를 Tauri **사이드카 프로세스**로
  번들(Node 바이너리 또는 bun 단일 실행파일화)한다. 프론트엔드 코드는 재사용.
- "지금 CLI냐 앱이냐"를 고민할 필요 없이, 뷰어=웹앱 / 분석기=별도 프로세스 구조면 나중에
  껍데기만 씌우면 된다.

## 4. 컴포넌트

### 4.1 분석기 (TS 플러그인)

각 모듈은 하나의 명확한 책임을 가지며 독립적으로 테스트 가능해야 한다.

| 모듈 | 역할 | 핵심 기술 |
|------|------|----------|
| 진입점 탐지기 (Entrypoint Detector) | "기능" 노드 추출. **Express 라우트**부터. 탐지기 인터페이스로 플러그인화하여 추후 Next.js/NestJS 추가 가능 | ts-morph AST |
| 호출 그래프 빌더 (Call Graph Builder) | "프로세스" = 함수→함수 호출 체인. 타입 체커로 호출 대상을 선언부로 해석 | TS type checker |
| 타입/모델 흐름 추출기 (Type Flow Extractor) | 각 함수의 입력/반환 타입, 건드리는 모델·DB 엔티티 식별 | 시그니처 + 휴리스틱 |
| 계층 빌더 (Hierarchy Builder) | 진입점 → 도달 함수 → 모듈/파일 그룹으로 묶어 드릴다운 트리 구성 | — |
| IR 직렬화기 (IR Serializer) | 위 결과를 그래프 IR(JSON)으로 출력 | — |

- 해석 불가한 동적 호출/심볼은 크래시 대신 `unresolved` 노드로 표시한다.
- 진입점 탐지기는 인터페이스 `EntrypointDetector`로 정의하고, Express 구현이 첫 구현체다.

### 4.2 그래프 IR (JSON 스키마)

언어 무관 보편 계약. `schemaVersion` 필드로 진화에 대비한다.

**노드**

```
{
  "id": "string",              // 안정적 고유 식별자
  "kind": "entrypoint" | "function" | "module" | "model",
  "label": "string",           // 예: "GET /users", "createUser", "User"
  "sourceLocation": { "file": "string", "line": number },
  "parentId": "string | null", // 계층(드릴다운) 부모. 최상위는 null
  "types": {                   // 선택. function 노드에서 사용
    "inputs": ["string"],
    "output": "string | null"
  }
}
```

**엣지**

```
{
  "source": "string",          // 노드 id
  "target": "string",          // 노드 id
  "kind": "call" | "contains" | "dataTouch",
  "meta": {                    // 선택. kind에 따라 의미 다름
    "access": "read" | "write",   // dataTouch일 때
    "dataType": "string"          // dataTouch일 때 흐르는 타입
  }
}
```

- `call` = 프로세스(제어 흐름): entrypoint→function, function→function
- `contains` = 계층(드릴다운): module→function, module→module
- `dataTouch` = 데이터 흐름: function→model (read/write 방향 + 타입)

**최상위 문서**

```
{
  "schemaVersion": 1,
  "language": "typescript",
  "nodes": [ ... ],
  "edges": [ ... ]
}
```

### 4.3 뷰어 (웹 앱)

- IR JSON 로드 → **최상위엔 진입점(entrypoint) 노드만** 표시 (크게 보기).
- 노드 클릭 → `contains` 엣지로 연결된 자식(함수/모듈)을 **펼치기** (semantic zoom / 드릴다운).
- 함수 노드엔 입력·반환 타입을 표시, `model` 노드와 `dataTouch` 엣지로 데이터 흐름 시각화.
- 노드 선택 시 소스 위치(`file:line`)를 표시한다. (실제 파일 점프는 Phase 2에서 Tauri로 구현)

## 5. 데이터 흐름 수준 (Phase 1)

**타입/모델 단위 흐름.** 진짜 데이터플로우(변수 값 추적)는 Phase 1 범위 밖이다.

- 각 함수 노드는 받는 타입·반환 타입을 노출한다.
- 모델/DB 엔티티는 `model` 노드로 표현하고, 이를 건드리는 함수에서 `dataTouch` 엣지를 그린다.
- "어떤 데이터가 이 경로를 지나가는가"를 시그니처 수준으로 파악하는 것이 목표.

## 6. 오류 처리

- 호출 대상 해석 실패(동적 호출, 외부 라이브러리 등): `unresolved` 처리, 분석 중단하지 않음.
- 파싱 불가 파일: 건너뛰고 경고를 누적해 IR 또는 stderr로 보고.
- 빈/무진입점 프로젝트: 유효한 빈 IR(`nodes: []`)을 출력.

## 7. 테스트 전략

IR이 계약이므로 분석기와 뷰어를 완전히 분리해 테스트한다.

- **분석기**: 픽스처 TS 프로젝트(작은 Express 앱 등) → IR JSON 스냅샷 테스트. 뷰어 없이 독립 검증.
- **뷰어**: 픽스처 IR JSON → 렌더/드릴다운 동작 검증. 분석기 없이 독립 검증.
- **IR 스키마**: 스키마 유효성 검증(예: JSON Schema/zod)으로 두 컴포넌트의 계약 일치를 보장.

## 8. Phase 1 범위 경계 (YAGNI)

**포함**

- TS/JS 대상
- Express 진입점 탐지
- 함수 호출 그래프 (프로세스)
- 타입/모델 단위 데이터 흐름
- 계층 드릴다운 웹 뷰어
- 버전드 그래프 IR 스키마

**제외 (나중 단계)**

- 멀티 언어 분석기 (Python/Go 등) — 단, IR/뷰어는 이를 수용하도록 설계
- Tauri 데스크톱 셸 — 단, 뷰어는 webview 호환으로 작성
- 진짜 데이터플로우(값 추적)
- Express 외 프레임워크 진입점
- 실시간 파일 워칭/증분 분석

## 9. 미해결/추후 결정 (구현 계획 단계에서 확정)

- 구체 기술 선택: 그래프 라이브러리(React Flow vs Cytoscape), 분석기 패키징(단일 레포 vs 워크스페이스),
  IR 검증 도구(zod vs JSON Schema).
- 노드 `id` 생성 규칙(파일 경로 + 심볼명 기반 등)의 정확한 포맷.
- 모델/DB 엔티티 탐지 휴리스틱의 구체 규칙(ORM 인식 범위 등).
